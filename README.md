# Day Assistant (LangGraph)

A daily-task assistant built as a **LangGraph state machine**, per the
project spec: natural-language understanding via the LLM, deterministic
CRUD via Python, no RAG, no external database.

## Architecture

```
START
  |
classify  ── Instructor + Pydantic (LLM, structured intent)
  |
  ├─ describe_day ─────► extract  (LLM: pulls tasks out of free text)
  ├─ list/complete/
  │  delete_todo ───────► select   (Python: loads the todo dict)
  ├─ add_todo ──────────► act
  └─ unknown ───────────► respond  (skips act entirely)
                              ▲
extract ──────────────────────┤
select ────────────────────────► act  (Python: the ONLY node that
                                        mutates TODO_STORE)
                                    │
                                    ▼
                                respond  (LLM: friendly wrapper only;
                                          list contents are always
                                          injected verbatim from Python,
                                          never generated by the model)
                                    │
                                    ▼
                                   END
```

| File          | Responsibility                                                   |
|---------------|-------------------------------------------------------------------|
| `config.py`   | Ollama connection + Instructor client setup                     |
| `schemas.py`  | Pydantic models for structured LLM output (`IntentClassification`, `ExtractedTasks`) |
| `state.py`    | `DayAssistantState` TypedDict, `TODO_STORE`, and CRUD helpers (business logic) |
| `nodes.py`    | The 5 graph nodes (classify, extract, select, act, respond)     |
| `graph.py`    | `StateGraph` wiring + conditional routing (pure orchestration)   |
| `main.py`     | Rich REPL entry point                                            |

**Golden rule enforced throughout:** the LLM understands language and
proposes structured data (intent, extracted tasks); only
`action_execution_node` in `nodes.py` ever reads or writes `TODO_STORE`.
Every LLM call is wrapped in `try/except` with a deterministic fallback,
so the REPL keeps working even if Ollama is offline.

## Setup

```bash
# 1. Make sure Ollama is running with the chat model this project uses
ollama pull gemma4:31b-cloud

# 2. Create a virtual environment
python3 -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt

# 4. Run it
python main.py
```

## Example session

```
You: Tomorrow I need to finish my AI assignment, call Ahmed, and buy milk.
Assistant: I've added three new tasks to your to-do list!

You: Show me my tasks.
Assistant: Here's your to-do list:
1. [ ] Finish AI assignment
2. [ ] Call Ahmed
3. [x] Buy milk
4. [ ] Finish AI assignment
5. [ ] Call Ahmed
6. [ ] Buy milk

You: Mark task 2 as completed.
Assistant: Task 2 ('Call Ahmed') is now marked as completed.

You: Add buy coffee filters to my list
Assistant: Added 'Buy coffee filters' to your to-do list.

You: Delete task 3
Assistant: Deleted task 3 ('Buy milk') from your to-do list.

You: exit
```

## Notes

- `TODO_STORE` is an in-process module-level dict (per spec) -- it resets
  every time you restart `main.py`. No SQLite, no vector DB.
- `complete_todo` / `delete_todo` parse a task number out of the message
  (e.g. "task 2"); if no number is present, they fall back to a
  case-insensitive title match (e.g. "mark call ahmed as done").
- This project was verified end-to-end with a mocked LLM backend to
  confirm routing, state merging, CRUD, and every fallback path behave
  correctly independent of Ollama being reachable.
